## 2. Year 1-2: SWE基盤構築フェーズ

### 基本方針

> MLのことは一旦忘れて、「信頼されるソフトウェアエンジニア」になることに集中する。
> ここで身につけるSWEの型が、将来MLEとしての最大の差別化要因になる。

### 2.1 業務で意識的に吸収すべきこと

#### テスト文化の内面化

| 学ぶこと | 具体的なアクション | 完了基準 |
|---------|----------------|--------|
| ユニットテストの書き方 | 業務コードでpytest/JUnitテストを毎回書く | テストなしでPRを出さなくなる |
| テスト設計手法 | 境界値分析、同値分割、状態遷移テストを理解する | テスト仕様書を一人で書ける |
| TDDの体験 | 小さな機能でTest-First開発を試す | 「テストから書く」が自然になる |
| モックの活用 | 外部API依存のテストでunittest.mock/pytestのfixtureを使う | DB・APIなしでテストが回せる |
| CI/CDでのテスト自動化 | GitHub Actions/Jenkinsでテストを自動実行する設定 | PRマージ前にテストが自動で走る |

#### コードレビューの技術

| 学ぶこと | 具体的なアクション |
|---------|----------------|
| レビューを**受ける**力 | 先輩のコメントを全てメモする。パターンを蓄積する |
| レビューを**する**力 | 後輩のPRをレビューする機会があれば積極的に担当する |
| レビュー観点の整理 | 可読性、保守性、パフォーマンス、セキュリティの4観点を意識 |
| 建設的なコメント | 「ここはダメ」ではなく「こうするとより良い理由は〜」と書く |

#### 設計力の獲得

| 学ぶこと | 具体的なアクション |
|---------|----------------|
| SOLID原則 | 単一責任、オープンクローズド、リスコフ置換、インターフェース分離、依存性逆転を業務コードで実践 |
| デザインパターン | Strategy, Factory, Observer, Decoratorを実際に使ってみる |
| クラス設計 | 「このクラスの責任は何か？」を常に問う |
| 設計ドキュメント | 技術仕様書・設計書を書く経験を積む |
| アーキテクチャ図 | C4モデル、シーケンス図、コンポーネント図を描けるようになる |

#### チーム開発のGit運用

| 学ぶこと | 具体的なアクション |
|---------|----------------|
| ブランチ戦略 | Git Flow / GitHub Flow / Trunk Based Developmentの違いを理解 |
| PRの書き方 | タイトル、説明、変更理由、テスト方法を明記する習慣 |
| コンフリクト解決 | rebase vs merge を理解し、適切に使い分ける |
| コミットメッセージ | Conventional Commitsの形式を身につける |

## 4. Year 3-5: MLE/DS転身フェーズ

### 基本方針

> SWE × ML の掛け算を武器に、社内異動 or 転職でMLE/DSポジションへ。
> この時点で「モデルを作れて、デプロイできて、運用できる」人材になっている。

### 4.1 深層学習スキルの深化

| スキル | 学習内容 | 方法 |
|-------|--------|------|
| **分散学習** | PyTorch DDP、FSDP、DeepSpeed ZeRO | マルチGPU環境での修士モデルの学習高速化 |
| **モデル最適化** | 量子化（INT8/INT4）、蒸留、プルーニング | ONNX Runtime、TensorRT、OpenVINO |
| **LLMファインチューニング** | LoRA、QLoRA、RLHF、DPO | HuggingFace PEFT + 小規模LLMでの実践 |
| **マルチモーダルAI深化** | Vision-Language Models、Audio-Text Fusion | 修士研究の発展として新たなモダリティを追加 |
| **Diffusion Models** | 画像生成、時系列生成 | 金融データの合成生成への応用 |
| **GNN（グラフNN）** | 銘柄間の関係性モデリング | 株式市場のネットワーク構造を学習 |

### 4.2 MLE特化スキル

| スキル | 内容 | 重要度 |
|-------|------|-------|
| **Feature Store** | Feast、Tecton。特徴量の管理・再利用・一貫性保証 | 高 |
| **モデルモニタリング** | データドリフト検知、性能劣化アラート、Evidently AI | 高 |
| **A/Bテスト基盤** | オンライン実験の設計・実行・分析 | 高 |
| **推論最適化** | バッチ推論 vs リアルタイム推論の設計判断、キャッシュ戦略 | 中 |
| **データパイプライン** | Apache Airflow / Prefect / Dagster でのワークフロー管理 | 中 |
| **ストリーミング処理** | Apache Kafka、Flink。リアルタイムデータ処理 | 低（必要になったら） |

### 4.3 DS特化スキル（DSを選ぶ場合）

| スキル | 内容 | 重要度 |
|-------|------|-------|
| **因果推論** | 施策効果の定量化。DID、傾向スコア、操作変数法 | 高 |
| **実験設計** | A/Bテストの設計、検出力分析、多腕バンディット | 高 |
| **ビジネスKPI設計** | 技術指標とビジネス指標の橋渡し | 高 |
| **ダッシュボード** | Streamlit、Gradio、Tableau/Power BI | 中 |
| **プレゼンテーション** | 非技術者への分析結果の説明力 | 高 |

## 付録: 判断基準メモ

### 「いつ転身すべきか」の判断基準

以下の条件が**3つ以上**満たされた時が転身のタイミング:

1. 富士通でのSWE経験が2年以上ある
2. テストを自然に書ける（テストカバレッジ70%以上の経験）
3. Docker + クラウドでのデプロイ経験がある
4. Kaggle/Signateでメダルを1つ以上持っている
5. 技術ブログの記事が20本以上ある
6. 社内でML関連の仕事に携わった実績がある
7. 転職先のMLE/DSポジションのJDを見て「自分にもできそう」と感じる

### 「富士通に残るべきか」の判断基準

以下の場合は残留が合理的:

- 社内にMLチームがあり、異動の見込みがある
- 年収・待遇に大きな不満がない
- 技術以外のスキル（マネジメント等）を積みたい
- 安定性を重視するライフステージ（結婚、住宅購入等）

### 「転職すべきか」の判断基準

以下の場合は転職が合理的:

- 社内にMLポジションがなく、異動の見込みもない
- 3年以上SWEをやり、ML側のスキルが錆び始めている
- 外部のMLE/DS市場での評価が社内評価を大きく上回る
- 技術的な成長機会が社内で限られている
